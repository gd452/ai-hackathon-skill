## 의사결정 (왜 이렇게 만들었나)

### Q: 왜 단일 스킬이 아닌 번들인가?

A: 교차검증 결과 — "AI 해커톤 = 단일 스킬 X, 번들 + playbook O". 80% 자동화 가능 영역 (문서·심사 보조·랭킹) 과 인간 영역 (장소·스폰서·갈등) 을 분리. 단일 스킬로 묶으면 책임 경계가 흐려진다.

### Q: 왜 외부 호출을 자제하나?

A: "Skill 이 외부 호출 / 셸 / 파일시스템 접근하면 보안 책임이 폭증한다." 이 스킬의 외부 호출 = **3 모델 LLM API 만**. 그 외 GitHub fetch · 정적 분석 등은 호출 환경의 다른 도구가 담당하는 구조. SKILL.md 에 호출 범위 명시.

### Q: 왜 "사람 심사 보조" 라고 매번 명시하나?

A: AI 채점이 단독 의사결정자가 되는 순간 (a) 시상 분쟁 시 책임 소재 모호 (b) 차별성 같은 정성 평가에서 hallucination 위험. **언어가 책임을 만든다** — "마켓플레이스" 같은 단어 하나가 법적 기대치를 형성한다.

### Q: 왜 한국어 기반인가?

A: 글로벌 AI 마켓 (GPT Store, Claude Skills 공식) 과의 정면 경쟁을 피하고 한국어 / 한국 운영 환경 niche 확보. 다국어는 후속 단계.

### Q: 가중치 기본값 (README 20% / 코드 25% / AI 25% / 완성도 20% / 차별성 10%) 의 근거는?

A: AI 활용도와 코드 품질을 가장 무겁게 (= 25% 씩) 가중. AI 시대 해커톤의 차별점은 "어떤 AI 도구를 어떻게 썼는가". 차별성을 10% 로 낮춘 이유는 정성 평가의 변동성 인식 — 가중치가 높으면 사람 심사위원 편차가 결과를 좌우.

### Q: 왜 채점 모델을 CLI 가 아닌 API 직접 호출로 바꿨나?

A: CLI 의존성 (Gemini CLI · Codex CLI + OAuth) 은 외부 사용자가 fork 후 5분 안에 동작시키는 데 장벽. 3 모델 모두 curl + 키 3 개로 통일하면 사전설치 = `brew install jq curl bun` 한 줄, 환경 변수 3 개 = 1 분.

## 협업 패턴

### Claude (Opus 4.7) 역할

- 주제 1차 정리 (brief)
- 종합 노트 작성 (synthesis)
- SKILL.md / 템플릿 / 예시 출력 / README 작성
- 시간선 + 의사결정 기록

### OpenAI (GPT-5.5) 역할

- 외부 시각 교차검증 — Claude 가 놓친 리스크·이견·신규이슈 도출
- "착수 순서 역전" 같은 큰 결정의 근거 제공

### 협업 방식

1. Claude 가 1차 정리 → self-contained brief 작성
2. OpenAI 에 background 로 발사 (read-only sandbox)
3. 응답 도착 후 Claude 가 합의·이견·신규이슈 종합
4. 결정 사항을 한국어 요약 + 미결 confirm

## 주장 vs 실제 구현 (Codex 2차 리뷰가 짚어준 표)

| 주장 | 실제 구현 | gap | 후속 |
|---|---|---|---|
| "AI 채점기" | rubric playbook + 3 모델 API 호출 | GitHub fetch · 정적 분석 미포함 | 호출 환경 도구 통합 명시 |
| "운영 80% 자동화" | 자료 생성 80% + 판정 보조 | "운영" 단어가 광범위 | 표현 보정 완료 |
| "데모 영상 평가" | transcript 입력 가정 | Whisper transcribe 미포함 | 호출 환경 도구가 transcript 제공 |
| "1차 정렬 보조" 명시 | 정확 | — | 유지 |
| 외부 호출 = LLM API 한정 | 정확 | — | 유지 (보안 장점) |

**핵심 인정**: 이 스킬은 "AI 채점기" 라기보다 **"채점 playbook + 자료 생성 자동화 + 3 모델 합의 점수"** 가 정확한 표현.
